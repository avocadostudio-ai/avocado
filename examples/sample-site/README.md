# sample-site

The simplest possible Avocado Studio integration: a Next.js 15 site whose
content lives in a single `content/pages.json` file. **No CMS, no API tokens,
no admin UI.** Useful as a starter, a smoke test for the SDK, or a reference
for what the higher-level CMS examples do under the hood.

Like the CMS examples, this one uses the **split-route** pattern: a fully
static published path plus a dynamic `/preview-draft` path that handles
editor and draft mode.

- **Site** (this app): http://localhost:3002
- **Content Studio** (Avocado editor): http://localhost:4100
- **Orchestrator**: http://localhost:4200

---

## 1. Prerequisites

```bash
# from the monorepo root, in a separate terminal
./start                        # boots orchestrator + editor + apps/site
# or, if you've already set up .env at the root:
pnpm dev
```

There is **nothing else to set up**. No env vars, no accounts, no tokens.

---

## 2. Run the site

```bash
pnpm --filter sample-site dev
```

Open http://localhost:3002 — you'll see the seeded home page (`/`) and the
seeded blog post (`/blog/building-with-blocks`). The pages and their blocks
come from `content/pages.json`.

---

## 3. Edit content

Open the Avocado **Content Studio** at http://localhost:4100:

1. Click the site picker → **Add site**:
   - **Site ID**: `sample-site`
   - **Origin**: `http://localhost:3002`
2. Start chatting. When you publish, the editor POSTs to
   `http://localhost:3002/api/editor/publish`, which uses the SDK's
   `createJsonFilePublishHandler` to:
   - Rewrite `content/pages.json` in place with the new pages.
   - Copy any inline image assets (e.g. AI-generated images) to
     `public/generated-images/` so they have stable public URLs.

You can also edit `content/pages.json` by hand and refresh.

## What's wired up

| File | Role |
|------|------|
| `app/[[...slug]]/page.tsx` | Static published route (`mode: "static"`) |
| `app/preview-draft/[[...slug]]/page.tsx` | Dynamic editor route (`mode: "preview"`) |
| `middleware.ts` | Rewrites editor/draft requests to the preview route |
| `app/api/editor/[...path]/route.ts` | Editor API + JSON-file publish handler |
| `lib/pages.ts` | Page lookup helpers reading from `content/pages.json` |
| `content/pages.json` | The pages — read at build, rewritten by the publish handler |

## Key SDK touchpoints

- `createSitePage({ mode: "static" | "preview", ... })` — `@ai-site-editor/site-sdk/page`
- `createEditorMiddleware()` — `@ai-site-editor/site-sdk/middleware`
- `createEditorApiHandler({ getPages, onPublish })` — `@ai-site-editor/site-sdk/routes`
- `createJsonFilePublishHandler(pagesPath, { publicDir })` — `@ai-site-editor/site-sdk/publish-handlers/json-file`

The page chrome (header + footer) is hand-coded inside `pages.json` rather than
auto-generated by `createSitePage`, so the example sets `chrome: false`.
The CMS examples take the opposite approach: they set `chrome: true` and let
the SDK build a header from `SiteConfig.navLabels` + slugs.

## What's *not* here

This example deliberately omits:
- An ISR revalidation webhook — there's no remote CMS to send one.
- `PUBLISH_TOKEN` protection on the publish endpoint — not relevant for local
  development on a JSON file. Add it for any deployment that's reachable
  from the public internet.
- Image hosting beyond `public/generated-images/`. AI-generated PNGs land
  there; if you want remote images, allowlist their hosts in
  `next.config.ts` `images.remotePatterns`.
