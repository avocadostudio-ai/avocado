# AI Website Editor PoC — Product + Project + Monorepo Spec

> Goal: Build a local Proof of Concept (PoC) where a non-technical user edits a Next.js website using a chat UI. The system applies changes instantly to a live preview (no rebuild), using structured edit operations generated by an LLM (OpenAI SDK first). Editor harness/chat must be separated from website code.

## 1) Product Vision

### What we’re building
A split-screen editor:
- Left: Chat-based editor (natural language)
- Right: Live website preview embedded in an iframe

The user can:
- Type "Add testimonials below hero"
- Instantly see the site update in preview (sub-second, no Vercel rebuild)
- Click a section in preview to “select” it and then type “make this shorter”
- Undo/redo changes safely
- (Publishing to GitHub/Vercel is out of scope for PoC)

### Key UX principles
- The preview is the canvas; chat is the control surface.
- Never show JSON to the user.
- Every assistant response must be short: summary + bullet changes + status.

## 2) Core Technical Idea (No rebuild live preview)

Edits do not modify React/TS files. They modify page composition data (JSON) consisting of:
- pages (`PageDoc`) with
- blocks (`BlockInstance`) referencing existing components by type + validated props.

Next.js renders pages from the JSON. When JSON changes, preview refreshes (`router.refresh` / refetch) instantly.

## 3) System Architecture (Local PoC)

We run three separate apps locally:

1. Website (Next.js) — renders published or draft content
2. Editor App (UI) — chat panel + iframe preview
3. Orchestrator (Fastify) — LLM calls + validation + applying operations + draft store + history

### Local Ports
- Website: `http://localhost:3000`
- Editor: `http://localhost:4100`
- Orchestrator: `http://localhost:4200`

## 4) Requirements (PoC Scope)

### Must-have
- Split-screen editor UI (chat + iframe preview)
- Page content stored as JSON (draft + published)
- Component registry with schema validation (Zod)
- LLM outputs strict JSON `EditPlan` (no free-form text)
- Apply changes atomically: validate plan → validate block props → apply ops → version++
- Instant preview update (no rebuild)
- Visual selection: clicking a block in iframe sets active block in editor context
- Undo/redo (session-scoped)

### Nice-to-have
- Highlight selected block in preview
- “Flash” updated block after change
- Model selector in editor UI (fast/balanced/reasoning)

### Out-of-scope for PoC
- GitHub commits / PRs / Vercel deployment
- Multi-user collaboration
- Authentication/authorization
- Creating new components via AI code gen
- Drag & drop editor

## 5) Data Model

### PageDoc
```ts
type PageDoc = {
  id: string
  slug: string          // e.g. "/", "/pricing"
  title: string
  updatedAt: string
  blocks: BlockInstance[]
}
```

### BlockInstance
```ts
type BlockInstance = {
  id: string            // unique per page, e.g. "b_hero_home"
  type: string          // must be one of allowed block types
  props: Record<string, any>
}
```

## 6) Component Registry + Validation

Website and orchestrator share the same registry and Zod schemas via a shared package.

Example allowed blocks (PoC):
- `Hero`
- `FeatureGrid`
- `Testimonials`
- `FAQAccordion`
- `CTA`
- `Card`
- `CardGrid`

Orchestrator validates any `add_block` or `update_props` patches against schemas before saving draft.

## 7) LLM Protocol: EditPlan

The LLM must output a single JSON object matching:

```ts
type EditPlan = {
  intent: "edit_plan" | "needs_clarification"
  summary_for_user: string
  change_log: string[]
  ops: Operation[]
}
```

Operations allowed:

```ts
type Operation =
  | { op: "create_page"; page: PageDoc }
  | { op: "add_block"; pageSlug: string; afterBlockId?: string; block: BlockInstance }
  | { op: "update_props"; pageSlug: string; blockId: string; patch: object }
  | { op: "remove_block"; pageSlug: string; blockId: string }
  | { op: "move_block"; pageSlug: string; blockId: string; afterBlockId?: string }
```

If user request is ambiguous, LLM returns:
- `intent: "needs_clarification"`
- `ops: []`
- `summary_for_user` contains 1–2 questions

## 8) Editor ↔ Site Integration (Separated codebases)

The Editor embeds the Site via iframe:
- `http://localhost:3000/pricing?__editor=1&session=dev&editorOrigin=http://localhost:4100`

The Site uses a minimal preview adapter package (enabled only when `__editor=1`) to:
- wrap blocks with `data-block-id` + `data-block-type`
- tag editable sub-fields within a block with `data-editable-target="<propPath>"` (e.g. `"title"`, `"items.0.heading"`)
- add overlay hover + click handling
- postMessage events to parent editor

Clicking a sub-field element sends `editablePath` alongside `blockId`, giving the orchestrator enough context to target a specific prop in an `update_props` patch without the user having to describe it.

To keep the website codebase clean, preview behavior lives in a separate workspace package (`packages/preview-adapter`) and the site consumes it through a minimal integration surface.

### postMessage Protocol (versioned)
All messages include:
- `protocol: "site-editor/v1"`
- `type`
- `payload`

#### Site → Editor
- `blockClicked`: `{ slug, blockId, blockType, editablePath?: string }` — `editablePath` is set when the click lands on a `data-editable-target` child element
- `routeChanged`: `{ slug }`

#### Editor → Site
- `highlightBlock`: `{ blockId, editablePath?: string }` — `editablePath` scrolls/highlights the specific sub-field after the block is focused
- `draftUpdated`: `{ focusBlockId?: string }`

### Instant refresh strategy (PoC)
No SSE needed initially. After orchestrator applies a plan successfully:
- Editor sends `draftUpdated` postMessage to iframe
- Site handles `draftUpdated` by calling `router.refresh()` or refetching JSON.
- If `focusBlockId` is present, site scrolls/highlights the target block after refresh for a smoother insert/update experience.

## 9) Orchestrator Responsibilities (Fastify)

### Storage (PoC)
In-memory maps:
- `publishedPages: Map<slug, PageDoc>`
- `draftPages: Map<session, Map<slug, PageDoc>>`
- `historyUndo: Map<session, Map<slug, PageDoc[]>>`
- `historyRedo: Map<session, Map<slug, PageDoc[]>>`
- `versions: Map<session, number>`

### Core pipeline
1. Receive user message + context
2. Build prompts with:
   - allowed block types + schema summary
   - current page doc (draft)
   - activeBlockId/type/editablePath (if set, LLM is constrained to target that block/field)
3. Call OpenAI model (chosen via `modelKey`)
4. Parse strict JSON to `EditPlan`
5. Validate plan structure
6. Validate block props
7. Apply ops atomically
8. Save draft + push undo history
9. Return: status + summary + changes + previewVersion (+ optional focusBlockId)

## 10) API Contracts (PoC)

### Content endpoints (used by Site)
- `GET /published/pages?slug=/pricing`
- `GET /draft/pages?session=dev&slug=/pricing`

### Editing endpoints (used by Editor)
- `POST /chat` — non-streaming
  - body: `{ session, slug, message, modelKey?, activeBlockId?, activeBlockType?, activeEditablePath? }`
  - `activeEditablePath` mirrors the `editablePath` from `blockClicked`; orchestrator uses it to constrain the LLM to a specific prop in `update_props`
  - response: `{ status, summary, changes, validationErrors?, suggestions?, previewVersion, focusBlockId?, modelUsed, modelKey, plannerSource }`

Optional streaming endpoint (Phase 2):
- `GET /chat/stream?...` — SSE streaming deltas then final
  - implemented shape:
    - `data: {"type":"status","message":"Planning edit..."}`
    - `data: {"type":"token","text":"<incremental text>"}` — zero or more progress tokens streamed while planning
    - `data: {"type":"final","result":<same object as POST /chat response>}`
    - `data: {"type":"error","result":<error payload>,"code":<http-like code>}`

### History endpoints
- `POST /history/undo` body `{ session, slug }`
- `POST /history/redo` body `{ session, slug }`

## 11) Model Selection (OpenAI SDK first)

Editor sends `modelKey` (not raw model id):
- `fast` | `balanced` | `reasoning` | `codex` (codex optional)

Orchestrator resolves to actual model strings via env vars:
- `OPENAI_MODEL_FAST=gpt-4o-mini`
- `OPENAI_MODEL_BALANCED=gpt-4o`
- `OPENAI_MODEL_REASONING=gpt-5.2`
- `OPENAI_MODEL_KEY=balanced`

## 12) Monorepo Setup (pnpm)

### Folder structure
```text
ai-site-editor-poc/
  pnpm-workspace.yaml

  apps/
    site/            # Next.js website
```

`pnpm-workspace.yaml`:
```yaml

  - "apps/*"
```

### Root scripts
- `pnpm dev` runs all 3 in parallel
- `pnpm dev:site`, `pnpm dev:editor`, `pnpm dev:orchestrator`

## 13) Implementation Milestones

### Milestone 1 (No LLM)
- Orchestrator: apply deterministic demo EditPlan rules (no LLM dependency)
- Editor: “Apply demo change” button
- Site: draft fetch + router.refresh on draftUpdated
- Block click events wired

### Milestone 2 (LLM)
- Add OpenAI integration to `/chat`
- Strict JSON parsing + validation
- Model selector dropdown in editor
- SSE streaming endpoint for chat progress (`GET /chat/stream`)

### Milestone 3 (Undo/Redo + polish)
- Undo/redo buttons
- Preview highlighting of selected block
- Updated-block flash
- Smooth insert/update focus using optional `focusBlockId` returned by orchestrator

### Current PoC Status
- Milestone 1: implemented
- Milestone 2: implemented (OpenAI path + deterministic fallback + SSE stream endpoint)
- Milestone 3: implemented baseline (undo/redo, highlight/flash, smooth focus on insert)

## 14) Success Criteria

The PoC is successful if:
- A user can create/modify a page via chat and see changes in preview immediately (< 500ms perceived)
- Clicking a section sets the context and “make this shorter” edits the right block
- Invalid changes are blocked with a helpful message
- Undo/redo works reliably

## 15) Engineering Principle

This is a deterministic editor:
- LLM proposes structured operations
- Orchestrator validates + applies
- Website renders JSON

LLM never directly edits code files.
